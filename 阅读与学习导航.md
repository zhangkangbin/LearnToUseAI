# AI 阅读与学习导航：从高效使用到构建可靠智能体系统

这份导航把仓库中的文章组织成两种读法：

- **递进主线**：从个人高效使用 AI 开始，逐步学到 Skill、MCP、函数调用、RAG、评估、安全和多智能体系统，适合系统学习。
- **快捷路线**：根据当前目标选择最短路径，先解决实际问题，再补齐相关知识，适合带着任务学习。

如果你第一次接触这些内容，沿着递进主线阅读；如果已经有明确目标，可直接跳到[快捷路线](#二按目标选择快捷路线)。无论选择哪条路线，都建议用同一个真实项目反复实践，而不是只阅读概念。

### 第一阶段：建立正确的 AI 使用习惯

首先阅读：[《普通人如何高效使用 AI——基础篇》](./普通人如何高效使用AI-基础篇.md)

- **学习目标**：学会描述背景、目标、约束和输出格式；理解为什么重要结果需要核验；建立隐私与安全意识。
- **推荐输出**：选择一个日常任务，例如会议纪要、邮件草稿或资料整理，保存一份可重复使用的提示词模板。
- **完成标准**：面对一个具体任务时，能独立写出清晰指令，获得可用结果，并知道如何检查事实、数字和敏感信息。

不要急着追求复杂工具。这个阶段最重要的能力，是把模糊需求变成清楚、可验证的任务。

### 第二阶段：把一次对话变成稳定流程

接着阅读：[《普通人如何高效使用 AI——进阶篇》](./普通人如何高效使用AI-进阶篇.md)

- **学习目标**：掌握任务拆解、分阶段协作、示例驱动、角色分工和结果迭代，让 AI 从“临时问答”升级为“流程伙伴”。
- **推荐输出**：把第一阶段的任务拆成“收集材料 → 分析 → 生成 → 审核 → 定稿”的工作流，并为每一步定义输入和输出。
- **完成标准**：同类任务重复执行时，结果结构基本稳定；出现问题时，能定位是输入、过程还是验收标准不清楚。

### 第三阶段：建立质量、评估与风险体系

然后阅读：[《普通人如何高效使用 AI——精通篇》](./普通人如何高效使用AI-精通篇.md)

- **学习目标**：从个人技巧提升到系统思维，理解上下文、知识、评估、权限、成本、可观测性和人机协作之间的关系。
- **推荐输出**：给自己的 AI 工作流增加检查清单、评分规则、失败处理方式和人工确认节点。
- **完成标准**：不再只判断“回答看起来不错”，而是能用明确标准衡量正确性、完整性、安全性和成本。

前三个阶段解决的是“如何正确使用 AI”。如果你还不准备开发自动化系统，到这里已经可以形成完整的个人学习闭环；需要让 AI 调用工具、连接数据或自动执行任务时，再进入第四阶段。

### 第四阶段：掌握 Skill、MCP 与函数调用

先看总览：[《AI 能力系统：Skill、MCP 与函数调用》](./ai-capability-system/README.md)，再按以下顺序学习：

1. [《技能与可复用流程》](./ai-capability-system/01-技能与可复用流程.md)：把经验、规则和步骤沉淀为可重复执行的能力。
2. [《MCP 与外部工具连接》](./ai-capability-system/02-MCP与外部工具连接.md)：理解模型如何通过统一协议发现和使用外部数据与工具。
3. [《函数调用与结构化操作》](./ai-capability-system/03-函数调用与结构化操作.md)：用结构化参数把模型意图转换成可校验、可执行的操作。
4. [《端到端项目周报案例》](./ai-capability-system/04-端到端项目周报案例.md)：把 Skill、MCP 和函数调用组合成一条完整工作流。

- **学习目标**：分清 Skill、MCP 和函数调用各自解决的问题，理解三者如何协作，而不是把它们当成三个相互替代的名词。
- **推荐输出**：实现或完整设计一个“项目周报助手”：读取提交记录，按项目归类，生成周报，校验结果，再保存或发送。
- **完成标准**：能说明每一步的输入、输出、工具权限和失败处理；知道哪些动作可自动执行，哪些动作必须由人确认。

## 三、每篇文章怎么学

建议每次只处理一个主题，并完成下面的闭环：

```mermaid
flowchart LR
    A["理解概念<br/>它解决什么问题"] --> B["阅读图示<br/>看清组件与数据流"]
    B --> C["跑通案例<br/>观察输入与输出"]
    C --> D["完成练习<br/>替换成自己的任务"]
    D --> E["复盘记录<br/>总结边界与失败原因"]
    E --> A
```

学习笔记至少回答五个问题：

1. 这个能力解决什么问题？不使用它会出现什么后果？
2. 它接收什么输入，产生什么输出，依赖哪些数据和权限？
3. 正常流程、失败流程和人工介入点分别在哪里？
4. 如何判断结果是正确、安全且值得信任的？
5. 它是否真的适合当前项目，还是增加了不必要的复杂度？

## 四、用一个项目贯穿全部学习

推荐使用“项目周报助手”作为贯穿案例，因为它既容易开始，又能逐步容纳高级能力：

| 学习阶段 | 给周报助手增加的能力 |
| --- | --- |
| 基础篇 | 根据明确模板生成周报草稿，并人工核对事实 |
| 进阶篇 | 拆分资料收集、分类、总结、审核和定稿步骤 |
| 精通篇 | 增加评分规则、隐私检查和发送前确认 |
| Skill | 固化周报格式、项目分类规则和表达风格 |
| MCP | 连接 Git 仓库、项目管理工具或文档系统 |
| 函数调用 | 用结构化参数读取提交、保存草稿或触发发送 |
| RAG 与记忆 | 检索项目背景、历史周报和用户偏好 |
| 评估与可观测性 | 检查遗漏、重复、事实错误并记录完整链路 |
| 安全与可靠性 | 控制权限，防止提示词注入，处理超时与重复发送 |
| 多智能体 | 在确有收益时拆分收集、撰写、核查和审批职责 |

每完成一个主题，只增加一项新能力并重新运行原有测试。这样可以直观看到复杂度从哪里产生，也能判断新技术是否带来了实际收益。

## 五、容易走偏的地方

- **跳过基础，直接搭复杂智能体**：需求和验收标准不清楚时，增加工具和智能体只会放大不确定性。
- **追逐工具名称，忽略问题本身**：先说明业务目标、数据来源和风险，再决定是否需要 Skill、MCP、RAG 或多智能体。
- **只阅读，不产出**：没有模板、流程图、接口定义、评估集或可运行项目，就很难形成可迁移能力。
- **只测成功路径**：真实系统还要测试空数据、错误参数、超时、权限不足、重复调用和恶意输入。
- **忽略评估与安全**：能够运行不等于可以上线；越接近真实业务，越要重视人工审批、最小权限和审计。
- **过早微调模型**：先尝试改进上下文、检索、工具、工作流和评估；只有证据表明模型适配仍是瓶颈时，再学习微调与蒸馏。

## 六、今天从哪里开始

- **第一次系统学习 AI**：从[基础篇](./普通人如何高效使用AI-基础篇.md)开始，完成一个真实的小任务后再继续。
- **已经经常使用聊天式 AI**：从[进阶篇](./普通人如何高效使用AI-进阶篇.md)开始，把一个常用任务改造成稳定流程。
- **会写提示词，想学习工具调用**：直接进入[AI 能力系统总览](./ai-capability-system/README.md)，按 Skill → MCP → 函数调用 → 案例的顺序学习。
- **已经开发过 AI 应用**：先用[高级 AI 系统学习地图](./advanced-ai-system/README.md)查漏补缺，再选择一条快捷路线完成项目。
- **准备上线真实业务**：优先走[生产可靠性、安全与治理路线](#路线-d生产可靠性安全与治理)，补齐评估、监控、权限、恢复和红队测试。

最好的学习顺序不是“把所有文章读完”，而是“理解一个概念，给真实项目增加一项能力，用证据确认它有效，再进入下一步”。
